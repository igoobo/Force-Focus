# RPA(명령어 자동 실행) 보안 패치 결과 보고서

## 1. 취약점 개요

데스크탑 에이전트의 `schedule.rs` 내부에는 서버 데이터 또는 내부 스토리지 정보에 기반하여 특정 애플리케이션을 예약된 시간에 강제로 실행하는 로직이 있습니다. 기존 코드에서는 실행될 애플리케이션의 경로(`exe_path`)와 인자(`args_str`)에 대해 아무런 검증 없이 `std::process::Command::new`를 사용하여 실행하고 있었습니다.

### 발견된 주요 문제점
- **임의의 코드 실행 취약점(RCE):** 검증되지 않은 실행 경로(`cmd.exe`,惡性스크립트 등)가 서버 혹은 해킹된 로컬 DB로부터 주입될 경우, 사용자 동의 없이 데스크탑 권한으로 실행될 수 있었습니다.
- **취약한 인자 파싱 기법:** `split_whitespace()`를 통해 단순히 공백 문자를 기준으로 인자를 분리하여, 띄어쓰기가 포함된 경로나 복잡한 특수문자가 포함된 인자를 정상적으로 처리할 수 없었으며, 명령어 주입(Command Injection) 공격의 빌미를 제공했습니다.

## 2. 해결 방안 및 패치 내용

위 문제점을 해결하기 위해 유효성 검증 계층을 마련하고, 인자 파싱 방식을 견고하게 개선했습니다.

### 2.1 절대 경로 및 `.exe` 확장자 검증
- 새로 도입된 `is_safe_executable_path` 함수 내에서 실행 파일 경로가 `절대 경로`인지 확인합니다.
- `path.exists()`를 통해 파일이 실제로 존재하는지 검증합니다.
- `.exe` 확장자의 프로그램만 제한적으로 실행하도록 하여, 다양한 우회용 스크립트 실행 공격(`.bat`, `.ps1`, `.vbs` 등)을 사전에 차단했습니다.

### 2.2 시스템 쉘 및 스크립트 엔진 블랙리스트 도입
기본 탑재된 위험한 시스템 쉘 프로그램을 명시적으로 차단하기 위해 다음 목록을 블랙리스트(`blacklist`)로 처리했습니다.
- `cmd.exe`, `powershell.exe`, `pwsh.exe`, `wscript.exe`, `cscript.exe`, `mshta.exe`, `bash.exe`, `wsl.exe`
- 위 파일들은 어떠한 경우에도 자동 스케줄러에 의해 강제로 실행될 수 없습니다.

### 2.3 따옴표(Quotes)를 고려한 안전한 인자 파서 구현
기존의 단순 공백 기준 분리 로직(`split_whitespace()`)을 제거하고 `parse_arguments` 함수를 추가로 구현했습니다.
- 이 함수는 명령어 인자 구성 시, 큰따옴표(`"`) 내에 포함된 공백은 인자 분리를 위한 공백으로 치환되지 않고 하나로 묶여 안전하게 래핑(Wrapping)되도록 보장합니다.

## 3. 적용 파일 위치
- `desktop-agent/src-tauri/src/managers/schedule.rs`

## 4. 향후 권고 사항
- **화이트리스트 기반 접근:** 이번 패치는 쉘 등을 막는 `블랙리스트` 위주의 처리입니다. 장기적으로는 사용자가 명시적으로 허용한 앱(예: Force-Focus를 통한 학습용 프로그램 등)만 실행할 수 있도록 진정한 `화이트리스트` 방식으로 전환하는 것을 권장합니다.
- **앱 서명(Signature) 검증 추가:** 실행하려는 프로그램의 인증서(Windows Authenticode)를 확인하여, 인증된 신뢰할 수 있는 개발사의 프로그램인지 검증하는 로직이 추가되면 보안 수준을 한층 더 높일 수 있습니다.
